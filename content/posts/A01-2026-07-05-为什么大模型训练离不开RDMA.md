## 一、引言：AI 训练的核心矛盾

大模型训练有一个核心矛盾：**算力越强，等数据的时间越长。**

以 GPT-4 级别的模型为例，训练需要数千张 GPU 协同工作。一张 H100 的计算速度是 2000 TFLOPS，但数据从 A 卡传到 B 卡，要走网络。网络传输速度远低于 GPU 的计算速度。

这个差距有多大？

```
GPU 算力：   2000 TFLOPS（每秒 2×10¹⁵ 次运算）
网卡带宽：   400 Gbps（每秒 5×10¹⁰ 字节）
```

这就意味着 GPU 大部分时间在**等数据**。

---

## 二、四种并行策略与通信模式

大模型训练不是简单地把模型放在一张卡上跑，而是把模型和数据拆分到多张 GPU 上。不同的拆分方式对应不同的通信模式。

### 1. 数据并行（Data Parallelism）

每张卡放一份完整的模型副本，但只处理不同的数据批次。

**通信模式：** 前向/反向传播结束后，AllReduce 同步梯度

**通信压力：** 随模型大小线性增长

### 2. 张量并行（Tensor Parallelism）

把单个矩阵运算切分到多张卡上。

**通信模式：** 每个 forward/backward 都需要 AllReduce

**通信压力：** **极高**，每次运算都伴随通信

### 3. 流水线并行（Pipeline Parallelism）

把模型的不同层放在不同的卡上，串行执行。

**通信模式：** P2P Send/Recv，只在层间边界通信

**通信压力：** 相对较低，但延迟敏感

### 4. MoE 并行（Mixture of Experts）

只有部分"专家"网络被激活，路由决定数据发给谁。

**通信模式：** All-to-All 通信

**通信压力：** **极高**，专家之间频繁通信

---

### 通信模式汇总

| 并行策略 | 通信模式 | 通信量 | 延迟敏感度 |
|---------|---------|:------:|:---------:|
| 数据并行 | AllReduce | 高 | 中 |
| 张量并行 | AllReduce | **极高** | **极高** |
| 流水线并行 | P2P Send/Recv | 低 | 高 |
| MoE 并行 | All-to-All | **极高** | 中 |

> 实际训练中，这四种并行策略往往**叠加使用**：数据并行 × 张量并行 × 流水线并行 × MoE。这意味着通信压力是乘法级别的增长。

---

## 三、通信瓶颈的具体数字

以一个 100B 参数的模型为例：

```
AllReduce 一次梯度同步 ≈ 400 GB 数据
如果使用 200 Gbps 网络：传输时间 ≈ 16 秒
如果使用 400 Gbps RDMA：传输时间 ≈ 8 秒
但一张 H100 跑一个 iteration 可能才 5 秒
```

这就意味着：**如果网络不够快，GPU 等通信的时间比计算的时间还长。**

---

## 四、为什么 TCP 不行？

传统数据中心走 TCP/IP，但 TCP 在大规模 AI 训练场景下有致命问题：

### 问题 1：CPU 开销大

TCP 收发数据需要 CPU 参与数据拷贝：

```
网卡 → CPU 内存（DMA）→ 内核 TCP 协议栈 → 
Socket buffer → 用户态应用程序 → 最终目的地
```

数据在内存里被**复制了至少 4 次**，每次复制都消耗 CPU 时间和内存带宽。

### 问题 2：延迟高

TCP 的内核协议栈增加了数十微秒的延迟。对于常规 Web 服务这没问题，但对于需要在毫秒级完成 AllReduce 的 AI 训练，这是不可接受的。

### 问题 3：吞吐受限

大规模集群中，TCP 的拥塞控制机制（慢启动、拥塞避免）在网络拓扑复杂的场景表现不佳，吞吐利用率往往只有 60-80%。

---

## 五、RDMA 如何解决

RDMA（Remote Direct Memory Access）的核心思想很简单：**让数据直接从 A 卡的内存到 B 卡的内存，绕过 CPU。**

```
传统 TCP：
A 卡内存 → A CPU → A 内核 → 网线 → B 内核 → B CPU → B 卡内存
                    ↑      ↑                    ↑      ↑
                   四次拷贝，CPU 全程参与

RDMA：
A 卡内存 → A RNIC → 网线 → B RNIC → B 卡内存
                ↑                    ↑
              零拷贝，CPU 不参与
```

### RDMA 三大优势

| 指标 | TCP | RDMA |
|-----|:---:|:----:|
| 数据拷贝次数 | 4 次 | 0 次（零拷贝）|
| CPU 卸载 | 否 | 是（由 RNIC 网卡完成）|
| 延迟（跨节点）| 50-100 μs | 1-3 μs |
| 吞吐利用率 | 60-80% | 95%+ |

### RDMA 的三种实现

目前在 AI 网络中最主流的是 **RoCEv2**（RDMA over Converged Ethernet v2）：

| 实现 | 特点 | AI 网络中的应用 |
|-----|------|:-------------:|
| **InfiniBand** | 专用网络，性能最强，成本最高 | NVIDIA DGX 标配 |
| **RoCEv2** | 基于以太网，成本低，生态好 | **主流 AI 集群** |
| iWARP | 基于 TCP，性能一般 | 较少用于 AI |

---

## 六、业界实践：RoCEv2 + NCCL

NVIDIA 的 NCCL（NVIDIA Collective Communications Library）是目前 AI 训练中**事实上的通信库标准**。它底层使用 RDMA：

```
用户态：NCCL 通信库
   ↓
系统态：libibverbs（RDMA verbs API）
   ↓
硬件层：RoCEv2 / InfiniBand RNIC
```

典型的 AI 集群拓扑：

```
┌─────────────┐    400 Gbps RoCE
│  GPU Server ├─────────────────┐
│  (8× H100)  │                 │
│  NCCL+RDMA  │                 │
└──────┬──────┘                 │
       │ PCIe Gen5              │
       │                        │
  ┌────┴────┐            ┌──────┴──────┐
  │ ConnectX│◄───────────│  ToR Switch │
  │  7 NIC  │  400Gbps   │  (无损网络) │
  └─────────┘            └─────────────┘
```

## 七、总结

**一句话回答标题的问题：**

> 大模型训练离不开 RDMA，因为它的并行策略（数据/张量/流水线/MoE）产生了**海量的通信需求**，传统 TCP 无法在 CPU 开销、延迟和吞吐上满足要求，而 **RDMA 的零拷贝、CPU 卸载和极低延迟**让它成为 AI 网络的必然选择。
