## 一、RDMA 核心对象

RDMA 编程涉及一组核心对象，它们共同构成了 RDMA 通信的基础。

### 1.1 PD — 保护域（Protection Domain）

**PD 是一个隔离边界。** 一个 PD 里可以包含多个 QP、多个 MR 和一个 CQ。不同 PD 之间的对象不能互相访问。类似进程地址空间的隔离概念，同一个应用程序可以创建多个 PD。

关系：

```
PD
├── QP1 (队列对 1)
├── QP2 (队列对 2)
├── MR A (内存区域 A)
├── MR B (内存区域 B)
└── CQ  (完成队列)
```

- **一个 PD 可以包含多个 QP 和多个 MR**（不是一对一的关系）

### 1.2 MR — 内存注册（Memory Region）

RDMA 要直接访问内存，必须先"注册"这块内存。注册后得到两个 Key：

| Key | 名称 | 用途 |
|-----|------|------|
| **lkey** | 本地 Key | 本地操作时验证 |
| **rkey** | 远程 Key | 对端操作时验证，发给对方 |

**为什么需要 MR？**
- RNIC（RDMA 网卡）直接 DMA 读写主机内存
- 必须告诉 RNIC 哪块内存是"合法的"
- rkey 发给对端，对端拿 rkey 才能读/写你的内存

### 1.3 QP — 队列对（Queue Pair）

每个 QP 包含两个队列：

- **SQ（Send Queue）**：发任务，往里扔 WQE
- **RQ（Recv Queue）**：收任务，往里扔 WQE

每个 QP 有一个唯一的 **QPN（QP Number）**，发包时填在 BTH 里告诉对端："这是发给你的 QP 的"。

**注意：** QP 并不绑定特定的 MR。同一个 QP 可以在不同的操作中使用不同的 MR（只要它们属于同一个 PD）。

### 1.4 CQ — 完成队列（Completion Queue）

CQ 存放操作完成的状态。SQ 和 RQ 的操作完成后，完成通知都放入 CQ，从 CQ 里 poll 出来就是 **WC（Work Completion）**。

- SQ 和 RQ 可以绑到**同一个 CQ**（简单，pingpong 常用）
- 也可以绑到**不同的 CQ**（性能场景，独立 poll）

### 1.5 WQ / WQE / WC

| 缩写 | 全称 | 含义 |
|------|------|------|
| **WQ** | Work Queue | 队列本身（SQ 或 RQ 的队列结构）|
| **WQE** | Work Queue Element | 队列里的每条任务，调用 `ibv_post_send()` 就把一个 WQE 扔进 SQ |
| **WC** | Work Completion | 完成通知，从 CQ 里 poll 出来 |

### 1.6 对象关系总结

```
Application
 │
 ├── ibv_open_device()  →  Context (ctx)
 │
 ├── ibv_alloc_pd()  →  PD
 │    ├── ibv_reg_mr()  →  MR (lkey + rkey)
 │    ├── ibv_create_qp()  →  QP（含 SQ + RQ）
 │    └── ibv_create_cq()  →  CQ
 │
 └── 通信流程：
      ibv_post_send(QP, WQE)  →  WQE 进 SQ
      ibv_poll_cq(CQ)         →  取出 WC
```

## 二、RoCEv2 报文结构

### 2.1 报文封装层次（从上到下逐层封装）

```
第 1 层：以太网帧头 (14B)
        MAC 目的地址 (6B) | MAC 源地址 (6B) | EtherType (2B)

第 2 层：IP 头 (20B)
        Version | TOS | Total Len | ID | Flags | TTL | Proto=UDP
        源 IP | 目的 IP

第 3 层：UDP 头 (8B)
        源端口（随机） | 目的端口 (4791) | 长度 | Checksum

第 4 层：BTH — 基础传输头 (12B)
        Opcode (1B) | 标志位+Pad (1B) | PSN (4B) | ... | QPN (4B)

第 5 层：扩展头（可选，如 RETH 16B / AETH 4B）

第 6 层：Payload — 实际传输的数据

第 7 层：ICRC (4B) — 不变校验和
        由发送端 RNIC 硬件计算，接收端 RNIC 校验
        覆盖 BTH ~ Payload 全部内容

第 8 层：FCS (4B) — 帧校验序列
        由每个交换机计算，只保链路，不保端到端
```

### 2.2 BTH 详解（Base Transport Header）

BTH 是 **12 字节**，RoCEv2 消息必须有的头部。

| Offset | Byte | 字段 | 说明 |
|--------|------|------|------|
| 0x00 | B0 | Opcode (8 bits) | 前 3 位：QP 类型（RC=0）；后 5 位：操作码 |
| 0x01 | B1 | Flags + Pad (8 bits) | S/M/L 标志位 + 填充字节数 |
| 0x02-05 | B2-B5 | PSN (32 bits) | 包序列号，RC 靠这个保证可靠有序 |
| 0x06-07 | B6-B7 | A-inv+A-Key (16 bits) | 主要用于原子操作 |
| 0x08-0B | B8-B11 | QPN (32 bits) | 目的 QP 编号 |

**Opcode 常见值：**

| Opcode | 含义 |
|--------|------|
| 0x00 | RC SEND First |
| 0x01 | RC SEND Middle |
| 0x02 | RC SEND Last |
| 0x08 | RDMA WRITE First |
| 0x09 | RDMA WRITE Middle |
| 0x0A | RDMA WRITE Last |
| 0x0C | RDMA READ Request |
| 0x0D | RDMA READ Response First |
| 0x0E | RDMA READ Response Last |
| 0x10 | Acknowledge (ACK) |

**Opcode 8 位结构：** 前 3 位 = QP 类型（000=RC, 001=UC, 100=UD），后 5 位 = 操作类型。

### 2.3 扩展头

根据操作类型不同，BTH 后面跟着不同的扩展头：

| 操作 | 扩展头 | 长度 | 携带内容 |
|------|--------|:----:|---------|
| SEND | 无 | 0B | 直接是 Payload |
| RDMA WRITE | RETH | 16B | 虚拟地址 + rkey + DMA 长度 |
| RDMA READ Request | RETH | 16B | 远端虚拟地址 + rkey + 长度 |
| RDMA READ Response | AETH | 4B | ACK 状态 + MSN |
| ACK / NAK | AETH | 4B | 确认/拒绝状态 |

**RETH（RDMA Extended Transport Header）结构：**

| Offset | 字段 | 说明 |
|--------|------|------|
| 0x00 | VA (8B) | 远端虚拟地址 |
| 0x08 | rkey (4B) | 远程 Key |
| 0x0C | DMA Len (4B) | 传输长度 |

### 2.4 UDP 端口 4791

- **目的端口：** 4791（固定，IANA 分配给 RoCEv2）
- **源端口：** 随机选取

**源端口随机的作用：** 交换机做 ECMP（等价多路径）负载均衡时，通过五元组 `(src_ip, dst_ip, src_port, dst_port, proto)` 计算哈希。如果所有流的源端口都一样，哈希结果会集中在同一条链路上。随机源端口让不同 QP 的流量散布到不同的物理路径上，提高整体带宽利用率。

### 2.5 ICRC vs FCS

| 校验 | 计算方 | 校验范围 | 作用 |
|------|--------|---------|------|
| **ICRC** | 端节点 RNIC 硬件 | BTH ~ Payload | **端到端**数据一致性 |
| **FCS** | 每个交换机 | 整个以太网帧 | **逐链路**校验 |

**ICRC 的存在意义：** 即使中间交换机重新计算 FCS 时引入了错误，接收端 RNIC 的 ICRC 校验也能检测出来。这是 RoCEv2 无损网络的核心保障之一。

## 三、RDMA 读与写的区别

### RDMA Read（单边读）

主动方发起 Read Request → 被动方 RNIC 直接读取内存返回数据 → 被动方 CPU 完全不知道发生了数据传输。需要知道被动方的 **rkey + 虚拟地址**。

### RDMA Write（单边写）

主动方直接把数据写到被动方的内存里 → 被动方 CPU 完全不知道被写了。需要知道被动方的 **rkey + 虚拟地址**。

### SEND/RECV（双边）

双方都要参与：接收端必须提前 post RECV WQE。不需要 rkey，不需要知道对端内存地址。更安全，但需要 CPU 参与。

### 对比总结

| 操作 | 别名 | CPU 参与 | 需要 rkey | 典型场景 |
|------|------|:--------:|:---------:|---------|
| **SEND/RECV** | 双边操作 | 双方都要 | ❌ | 小消息、控制面 |
| **RDMA WRITE** | 单边写 | 只有发起方 | ✅ | 梯度同步（NCCL）|
| **RDMA READ** | 单边读 | 只有发起方 | ✅ | 参数拉取 |

## 四、本地练习记录

```c
#include <infiniband/verbs.h>

// 1. 创建 Context（使用 rxe0）
struct ibv_context *ctx = ibv_open_device(ib_dev);

// 2. 创建 PD
struct ibv_pd *pd = ibv_alloc_pd(ctx);

// 3. 注册 MR（获得 lkey + rkey）
char *buf = malloc(BUF_SIZE);
struct ibv_mr *mr = ibv_reg_mr(pd, buf, BUF_SIZE,
    IBV_ACCESS_LOCAL_WRITE | IBV_ACCESS_REMOTE_WRITE | IBV_ACCESS_REMOTE_READ);

// 4. 创建 CQ
struct ibv_cq *cq = ibv_create_cq(ctx, 128, NULL, NULL, 0);

// 5. 创建 QP（当前状态：RESET）
struct ibv_qp_init_attr qp_attr;
memset(&qp_attr, 0, sizeof(qp_attr));
qp_attr.send_cq = cq;
qp_attr.recv_cq = cq;  // 也可以绑不同的 CQ
qp_attr.qp_type = IBV_QPT_RC;
// ... 其他初始化参数
struct ibv_qp *qp = ibv_create_qp(pd, &qp_attr);
// QP 编号 = qp->qp_num
```

已完成：
1. ✅ 创建 Context（`ibv_open_device`）
2. ✅ 创建 PD（`ibv_alloc_pd`）
3. ✅ 注册 MR（`ibv_reg_mr`，获得 lkey + rkey）
4. ✅ 创建 CQ（`ibv_create_cq`）
5. ✅ 创建 QP（`ibv_create_qp`，当前状态：**RESET**）

## 五、小结

| 对象 | 作用 | 关键点 |
|------|------|--------|
| PD | 隔离域 | 一个 PD 可圈住多个 QP + 多个 MR |
| MR | 内存注册 | lkey 本地 / rkey 远程 |
| QP | 收发队列对 | SQ + RQ，不绑定特定 MR |
| CQ | 完成队列 | SQ/RQ 共用或分开，poll 出 WC |
