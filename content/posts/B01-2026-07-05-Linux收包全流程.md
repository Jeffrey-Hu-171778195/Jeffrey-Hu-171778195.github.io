## 一、引言

当你在浏览器里输入一个 URL 按下回车，数据包从网卡到浏览器进程，中间经历了什么？

一条完整的收包路径：

```
网卡 → DMA → 内核内存 → 硬中断 → 软中断(NAPI) →
内核协议栈(L2/L3/L4) → Socket 缓冲区 → 用户进程
```

本文逐层拆解每个环节。

## 二、网卡层：硬件接收

### 2.1 网卡收包

数据包到达网卡后，网卡通过 **DMA（Direct Memory Access）** 直接将数据写入系统内存中的 **Ring Buffer（环形缓冲区）**，全程不需要 CPU 参与。

```
┌─────────┐     DMA      ┌──────────────┐
│  网卡    │ ──────────→  │ Ring Buffer  │
│  收到包  │              │ （内核内存） │
└─────────┘              └──────────────┘
```

### 2.2 硬中断（Hard IRQ）

DMA 完成后，网卡触发 **硬中断** 通知 CPU："数据到了"。

CPU 立即暂停当前工作，执行中断处理程序（Interrupt Handler），但这里只做最必要的事：

1. 记录收到新包
2. **触发软中断（SoftIRQ）** 把后续工作交给下半部
3. 快速返回

为什么要分两步？因为中断处理期间 CPU 不响应其他中断，如果直接在这里处理整个协议栈，系统就"卡死"了。

## 三、NAPI：收包的效率机制

**NAPI（New API）** 是 Linux 收包的核心机制，解决了一个经典问题：**高流量下中断风暴导致 CPU 打满**。

### NAPI 的工作原理

```
传统方式（纯中断）：
收包 → 中断 → 处理 → 收包 → 中断 → 处理 → ...
           ↑ 包越多，中断越多，CPU 越忙

NAPI 方式（中断+轮询）：
收包 → 中断 → 切换到轮询模式 →
       批量收包 → 批量收包 → ...
       没有新包了 → 切回中断模式
```

### NAPI 的流程

1. 收到第一个包，触发硬中断
2. 网卡驱动注册的 NAPI poll 函数被调度到软中断上下文
3. 关闭网卡中断（避免继续中断）
4. 在软中断中循环批量读取 Ring Buffer 中的包
5. 每批读取完毕后交给协议栈处理
6. 当 Ring Buffer 为空时，重新开启中断，等待下一个包

### 关键参数

```
gro_flush_timeout    ← 控制轮询超时
netdev_budget        ← 每次轮询最多处理多少个包（默认 300 个）
```

## 四、内核协议栈处理

从网卡驱动拿到 skb（Socket Buffer）后，协议栈逐层处理：

### 4.1 L2 — 数据链路层（以太网）

```
skb → 检查 MAC 地址 → 不是我的？丢弃
    → 检查 EtherType（IPv4/IPv6/ARP）
    → 取下 VLAN 标签
    → 交给对应的上层协议处理函数
```

此时关键功能：
- **GRO（Generic Receive Offloading）**：将多个小包合并成一个大包，减少协议栈处理次数
- **RPS（Receive Packet Steering）**：将收包负载均衡到多个 CPU 核心

### 4.2 L3 — 网络层（IP）

```
ip_rcv() 进入 IP 协议处理 →
    检查 IP 头校验和
    检查 TTL，TTL=0 则丢弃
    检查 IP 地址是否匹配
    处理 IP 分片重组（如果需要）
    查找路由表，决定本地交付还是转发
    如果是本地交付，交给 L4
```

如果是转发包，还会涉及 `netfilter` 钩子点（iptables/nftables）。

### 4.3 L4 — 传输层（TCP/UDP）

**TCP 收包路径：**

```
tcp_v4_rcv() →
    查找对应的 socket（通过四元组：src_ip:port → dst_ip:port）
    检查 TCP 序列号
    处理 TCP 状态机
    TCP 重组（乱序包放入 ofo 队列）
    确认（ACK）
    将数据放入 socket 接收缓冲区
    唤醒等待的进程
```

**UDP 收包路径：**

```
udp_rcv() →
    查找对应的 socket
    直接放入 socket 接收队列
    如果 socket 缓冲区满，丢弃包（累计 UDP 丢包计数）
```

## 五、用户态：数据到达进程

```c
// 用户进程调用 recvfrom()
ssize_t recvfrom(int sockfd, void *buf, size_t len, int flags,
                struct sockaddr *src_addr, socklen_t *addrlen);
```

调用路径：

```
recvfrom() → 系统调用 →
    sock_recvmsg() → tcp_recvmsg()/udp_recvmsg() →
    从 socket 接收缓冲区拷贝数据到用户空间
    如果缓冲区为空，进程休眠（阻塞模式）
        或返回 EAGAIN（非阻塞模式）
```

### 数据拷贝次数

```
网卡 DMA → Ring Buffer（第 1 次，无 CPU 参与）
Ring Buffer → skb（第 2 次，协议栈访问）
skb → Socket 缓冲区（第 3 次）
Socket 缓冲区 → 用户空间（第 4 次，系统调用）
```

**如果使用 mmap + 零拷贝技术，可以减少到 1-2 次。**

## 六、完整路径总结

```
                    ┌──────────────────────┐
                    │    用户进程           │
                    │    recvfrom()         │
                    └──────────┬───────────┘
                               │ 系统调用
                    ┌──────────▼───────────┐
                    │    Socket 缓冲区      │  L4
                    │    TCP 重组/状态机    │
                    └──────────┬───────────┘
                               │
                    ┌──────────▼───────────┐
                    │    IP 层             │  L3
                    │    路由/分片重组      │
                    └──────────┬───────────┘
                               │
                    ┌──────────▼───────────┐
                    │    以太网层           │  L2
                    │    GRO/RPS/MAC 过滤   │
                    └──────────┬───────────┘
                               │
                    ┌──────────▼───────────┐
                    │    NAPI 轮询          │  软中断
                    │    Ring Buffer 读取   │
                    └──────────┬───────────┘
                               │
                    ┌──────────▼───────────┐
                    │    硬中断 + DMA      │  硬件
                    │    网卡接收          │
                    └──────────────────────┘
```
